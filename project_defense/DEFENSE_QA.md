## 1. Общее об архитектуре

---

**В1. Что представляет собой проект? Опишите его в двух словах.**

Spotify Clone — веб-приложение для воспроизведения музыки. Пользователи слушают треки, создают плейлисты. Backend написан на FastAPI с PostgreSQL, фронтенд — на React. Всё упаковано в Docker-контейнеры.

---

**В2. Какой технологический стек используется на бэкенде?**

- **FastAPI** — веб-фреймворк (Python 3.11)
- **PostgreSQL 16** — база данных
- **SQLAlchemy 2.0** — ORM
- **Alembic** — миграции БД
- **python-jose** — JWT-токены
- **bcrypt** — хеширование паролей
- **Uvicorn** — ASGI-сервер
- **Docker + Docker Compose** — контейнеризация
- **Nginx** — обратный прокси для фронтенда

---

**В3. Почему выбрали FastAPI, а не Flask или Django?**

FastAPI дает три ключевых преимущества: автоматическую документацию (Swagger/OpenAPI из кода без усилий), встроенную валидацию данных через Pydantic и поддержку async/await для WebSocket. Django слишком тяжёлый для API-проекта, Flask требует много плагинов вручную. FastAPI — современный, быстрый и типобезопасный.

---

**В4. Как устроена структура бэкенда?**

```
app/
├── main.py          — точка входа, регистрация роутеров, CORS, /health
├── config.py        — настройки (JWT, БД, admin-emails) из .env
├── database.py      — движок SQLAlchemy, get_db dependency
├── dependencies.py  — get_current_user_id, get_admin_user_id
├── schemas.py       — Pydantic-схемы (валидация входа/выхода)
├── utils.py         — bcrypt, JWT (create/decode)
├── websocket.py     — ConnectionManager
├── models/          — ORM-модели (таблицы БД)
└── routes/          — эндпоинты по доменам (auth, songs, albums…)
```

---

**В5. Как организовано разделение по сервисам?**

Каждый домен вынесен в отдельный роутер: `auth`, `songs`, `albums`, `playlists`, `search`, `users`, `player`, `recommendations`, `upload`, `seed`, `websocket`. Каждый роутер монтируется на свой префикс (`/api/songs`, `/api/auth` и т.д.) в `main.py`. Это позволяет легко масштабировать — в будущем каждый роутер можно вынести в отдельный микросервис.

---

**В6. Сколько таблиц в базе данных? Перечислите.**

8 таблиц:

1. `auth_users` — учётные данные (email, password_hash)
2. `user_profiles` — публичный профиль (username, avatar, bio)
3. `albums` — музыкальные альбомы
4. `tracks` — треки (связаны с альбомом через album_id)
5. `playlists` — плейлисты пользователей
6. `playlist_tracks` — связующая таблица плейлист↔трек с полем `position`
7. `listening_history` — история прослушиваний
8. `user_statuses` — онлайн-статус и текущая активность (для WebSocket)
9. `messages` — сообщения чата

---

**В7. Почему пользователь разделён на две таблицы — `auth_users` и `user_profiles`?**

Разделение по ответственности (Single Responsibility). `auth_users` хранит только чувствительные данные для входа (email, хеш пароля) — они нужны системе безопасности. `user_profiles` — публичные данные (username, аватар, bio) — нужны остальным сервисам. Если нужно показать имя пользователя другим, мы отдаём профиль, не раскрывая email. Плейлисты и история прослушиваний ссылаются на `user_profiles.id`, а не на `auth_users.id`.

---

**В8. Как работает связь между таблицами? Какие есть внешние ключи?**

- `user_profiles.id` → `auth_users.id` (CASCADE DELETE, 1:1)
- `tracks.album_id` → `albums.id` (SET NULL при удалении альбома — трек становится синглом)
- `playlists.owner_id` → `user_profiles.id` (CASCADE DELETE)
- `playlist_tracks.playlist_id` → `playlists.id` (CASCADE DELETE)
- `playlist_tracks.track_id` → `tracks.id` (CASCADE DELETE)
- `listening_history.user_id` → `user_profiles.id` (CASCADE DELETE)
- `listening_history.track_id` → `tracks.id` (CASCADE DELETE)

---

**В9. Почему при удалении альбома треки не удаляются, а остаются как синглы?**

Намеренное решение: если администратор удалил альбом, треки не должны исчезать из плейлистов слушателей. Поэтому `album_id` в треке помечен `ondelete="SET NULL"` — при удалении альбома поле становится NULL, трек остаётся в системе как сингл. Это отличается от плейлистов, где при удалении — каскадное удаление.

---

**В10. Что такое `/api/seed` и зачем он нужен?**

Эндпоинт для наполнения базы тестовыми данными: создаёт тестового пользователя (`test@example.com / test123`), 4 альбома, 18 треков с аудио с SoundHelix (бесплатные mp3-примеры) и тестовый плейлист. Используется при первом запуске для демонстрации. Параметр `?force=true` обновляет URL треков без пересоздания всего.

---

**В11. Есть ли в проекте API-документация?**

Да, автоматически генерируется FastAPI. После запуска доступна по адресам:

- `http://localhost:8000/docs` — Swagger UI (интерактивная)
- `http://localhost:8000/redoc` — ReDoc

Документация строится из Pydantic-схем и аннотаций типов Python.

---

**В12. Как проверить, что backend работает?**

Эндпоинт `GET /health` выполняет `SELECT 1` к PostgreSQL и возвращает `{"database": "ok"}`. Также у каждого роутера есть свой `GET .../health` с именем сервиса. Docker Compose настроен с `healthcheck` для Postgres — backend не стартует, пока БД не готова.

## 2. FastAPI и устройство API

---

**В13. Что такое Dependency Injection в FastAPI? Как вы его используете?**

Dependency Injection (DI) — механизм, когда FastAPI сам создаёт и передаёт нужные объекты в функцию-обработчик. В проекте три основные зависимости:

- `get_db` — создаёт SQLAlchemy-сессию на время запроса, закрывает после
- `get_current_user_id` — извлекает user_id из JWT, возвращает 401 если нет токена
- `get_admin_user_id` — проверяет, что email пользователя в списке ADMIN_EMAILS, иначе 403

Используются через `Depends(...)` в сигнатуре функции.

---

**В14. Как работает `get_db`? Почему используется `yield`?**

```python
def get_db():
    db = SessionLocal()
    try:
        yield db      # сессия передаётся в обработчик
    finally:
        db.close()    # выполняется ВСЕГДА после ответа
```

`yield` превращает функцию в генератор — FastAPI понимает это как контекстный менеджер. Сессия создаётся до запроса, передаётся обработчику, а после отправки ответа (даже при исключении) закрывается в `finally`. Это гарантирует отсутствие утечек соединений.

---

**В15. Что такое Pydantic и зачем нужны схемы?**

Pydantic — библиотека валидации данных. Схемы (классы, унаследованные от `BaseModel`) описывают структуру входящих и исходящих данных. FastAPI автоматически:

- десериализует JSON из тела запроса в объект схемы
- валидирует типы (например, `EmailStr` проверит формат email)
- сериализует объект схемы обратно в JSON для ответа
- генерирует документацию

Если данные не соответствуют схеме — автоматически 422 Unprocessable Entity.

---

**В16. Чем `TrackCreate` отличается от `TrackUpdate`?**

`TrackCreate` наследует `TrackBase` — все поля обязательны (кроме явно Optional). `TrackUpdate` — все поля Optional (могут быть None). В обработчике `PUT` используется `model_dump(exclude_unset=True)` — обновляются только переданные поля, остальные не трогаются. Это реализует паттерн PATCH-семантики через PUT.

---

**В17. Что такое `response_model` и зачем он нужен?**

`response_model` в декораторе роута говорит FastAPI: «Отфильтруй ответ через эту схему». Это нужно для:

1. **Безопасности** — случайно не вернуть лишние поля (например, `password_hash`)
2. **Документации** — Swagger покажет правильную структуру ответа
3. **Сериализации** — преобразует ORM-объект в JSON

---

**В18. Почему на некоторых роутах два декоратора — `@router.get("")` и `@router.get("/")`?**

Это исправление конкретного бага: FastAPI + Nginx в Docker выдавал 307-редирект, когда запрос приходил без слэша (`/api/search`) на эндпоинт с путём `"/"`. Клиент после редиректа терял тело запроса. Решение — зарегистрировать оба варианта: с пустой строкой `""` и со слэшем `"/"`. Тогда оба URL обрабатываются напрямую без редиректа.

---

**В19. Как работает пагинация треков?**

Принимает параметры `limit` (макс. 100, по умолчанию 50) и `offset` через `Query`. Вместо `.limit().offset()` используется `.slice(start, stop)` — это более безопасный способ для PostgreSQL, который не падает на граничных значениях. Треки сортируются по `created_at DESC` (новые первыми).

---

**В20. Как устроена авторизация администратора?**

В `config.py` есть переменная `ADMIN_EMAILS` — set из email-адресов, считанных из переменной окружения. Функция `get_admin_user_id` сначала проверяет JWT (получает user_id), потом делает запрос в БД, достаёт email пользователя и проверяет, есть ли он в `ADMIN_EMAILS`. Если нет — 403 Forbidden. Это простая, но рабочая система ролей без отдельной таблицы.

---

**В21. Что возвращает `/api/search`? Как он устроен?**

Принимает параметр `q` (строка минимум 1 символ). Выполняет три запроса к БД:

1. Треки: фильтр по `title`, `artist`, `album_name` через `OR + ilike` (регистронезависимо)
2. Альбомы: фильтр по `title`, `artist`
3. Артисты: уникальные имена из результатов треков и альбомов, где artist содержит запрос

Возвращает `{"tracks": [...], "albums": [...], "artists": [...]}`.

---

**В22. Как обрабатываются ошибки в API?**

Через `HTTPException` с кодом статуса и описанием:

- 400 — некорректные данные (email уже занят)
- 401 — нет/невалидный токен
- 403 — нет прав (не админ)
- 404 — объект не найден
- 422 — ошибка валидации Pydantic (автоматически)
- 500 — непредвиденная ошибка сервера (в seed — перехватывается явно с rollback)

## 3. Аутентификация и JWT

---

**В23. Как работает регистрация пользователя?**

1. Клиент отправляет `POST /api/auth/register` с `email`, `username`, `password`
2. Проверяется уникальность email в `auth_users` и username в `user_profiles`
3. Пароль хешируется через bcrypt: `get_password_hash(password)`
4. Создаётся запись `AuthUser` (email + hash)
5. Вызывается `_ensure_profile` — создаётся `UserProfile` с username
6. Генерируются access + refresh токены
7. Возвращается `{"access_token": "...", "refresh_token": "...", "user": {...}}`

---

**В24. Как работает вход (login)?**

1. `POST /api/auth/login` с `email` и `password`
2. Ищем пользователя по email в `auth_users`
3. Сравниваем пароль с хешем через `verify_password` (bcrypt.checkpw)
4. Если не совпадает — 401 «Invalid email or password»
5. Если ок — генерируем access + refresh токены и возвращаем вместе с данными пользователя

---

**В25. Что такое JWT? Из чего он состоит?**

JWT (JSON Web Token) — это подписанная строка из трёх частей, разделённых точками:

- **Header** — алгоритм (`{"alg": "HS256", "typ": "JWT"}`)
- **Payload** — данные: `sub` (user_id), `email`, `exp` (время истечения), `type` (access/refresh)
- **Signature** — HMAC-SHA256 подпись (header + payload + SECRET_KEY)

Подпись позволяет серверу проверить токен без обращения к БД.

---

**В26. Зачем нужны два токена — access и refresh?**

- **Access token** — короткоживущий (30 минут). Используется для каждого запроса. Если украден, злоумышленник имеет доступ максимум 30 минут.
- **Refresh token** — долгоживущий (7 дней). Хранится в клиенте. Используется только для получения нового access token когда тот истёк. Это баланс между безопасностью и удобством.

---

**В27. Как backend проверяет токен на каждом запросе?**

1. FastAPI через `HTTPBearer` извлекает токен из заголовка `Authorization: Bearer <token>`
2. Вызывается `decode_token` из `utils.py` — `jwt.decode(token, SECRET_KEY, algorithms=["HS256"])`
3. Если токен невалиден или истёк — `JWTError`, возвращается `None`
4. Из payload достаётся `sub` (user_id) — это UUID пользователя
5. Возвращается `UUID(user_id)` — этот ID используется в обработчиках

---

**В28. Где хранится SECRET_KEY? Насколько это безопасно?**

В переменной окружения `SECRET_KEY`, которая задаётся в `.env` файле или в `docker-compose.yml`. В коде (`config.py`) есть fallback: `secrets.token_urlsafe(32)` — если ключ не задан, генерируется случайный. Это небезопасно для production (ключ меняется при рестарте), поэтому в production ВСЕГДА надо задавать фиксированный SECRET_KEY.

---

**В29. Что такое bcrypt и почему он лучше MD5/SHA-256 для паролей?**

bcrypt — алгоритм хеширования паролей, специально замедленный. Его ключевые особенности:

1. **Встроенная соль** — каждый хеш уникален, даже для одинаковых паролей. Радужные таблицы не работают.
2. **Настраиваемая сложность (cost factor)** — вычисление хеша занимает время (~100ms), что делает перебор нереальным.
3. MD5/SHA-256 — быстрые (миллиарды в секунду на GPU), не подходят для паролей.

---

**В30. Как работает `verify_password`? Что значит prefixcheck `oauth:`?**

```python
def verify_password(plain_password, hashed_password):
    if hashed_password.startswith("oauth:"):
        return False  # OAuth-пользователи не могут входить по паролю
    return bcrypt.checkpw(plain_password.encode(), hashed_password.encode())
```

Если пользователь зарегистрировался через OAuth (например, Google), в поле `password_hash` хранится маркер `oauth:...` вместо bcrypt-хеша. Попытка войти по паролю всегда вернёт False — нельзя взломать OAuth-аккаунт перебором пароля.

---

**В31. Как обновить access token, когда он истёк?**

Клиент отправляет `POST /api/auth/refresh` с телом `{"refresh_token": "..."}`. Backend:

1. Декодирует refresh token
2. Проверяет поле `type == "refresh"` в payload
3. Достаёт user_id, ищет пользователя в БД
4. Генерирует новую пару access + refresh токенов

---

**В32. Что произойдёт, если передать access token вместо refresh token в `/refresh`?**

Проверка `payload.get("type") != "refresh"` вернёт 401 — access token имеет `type: "access"`, не `"refresh"`. Подмена невозможна.

---

**В33. Есть ли в проекте blacklist токенов (инвалидация)?**

Нет — это намеренно для MVP. Реализована только проверка подписи и времени жизни. В endpoint `/logout` комментарий: «В production здесь нужно инвалидировать токен». В production решением был бы Redis-список отозванных токенов (или хранение версии токена в БД).

---

**В34. Что такое `get_optional_user_id`? Когда он используется?**

Функция-зависимость, которая возвращает `UUID` пользователя если токен есть и валиден, и `None` если токена нет (не бросает исключение). Используется в эндпоинтах, доступных и авторизованным и анонимам, но поведение разное: например, в `GET /api/playlists/` авторизованный видит свои приватные плейлисты + публичные, аноним — только публичные.

---

**В35. Как `_ensure_profile` помогает при создании пользователя?**

При регистрации `AuthUser` создаётся сразу, а `UserProfile` создаётся через `_ensure_profile`. Функция ищет профиль по user_id: если не найден — создаёт из `initial_username`. Если username занят — добавляет суффикс `_1`, `_2` и т.д. Это решает проблему race condition при параллельной регистрации с одинаковым username и позволяет создать профиль «лениво» при первом обращении.

## 4. База данных и SQLAlchemy

---

**В36. Что такое SQLAlchemy ORM? Почему не писали SQL вручную?**

SQLAlchemy ORM позволяет работать с таблицами как с Python-классами. Вместо `SELECT * FROM tracks WHERE id = $1` пишем `db.query(Track).filter(Track.id == track_id).first()`. Преимущества: защита от SQL-инъекций, переносимость между БД, читаемость кода, удобная работа с отношениями (relationship).

---

**В37. Что такое `pool_pre_ping=True` в настройке движка?**

При долгом простое PostgreSQL закрывает соединения. Если приложение возьмёт «мёртвое» соединение из пула — получит ошибку. `pool_pre_ping=True` заставляет SQLAlchemy перед выдачей соединения из пула делать `SELECT 1` и при ошибке переподключаться. Это делает приложение устойчивым к временным разрывам.

---

**В38. Что такое `relationship` и `cascade="all, delete-orphan"`?**

`relationship` определяет связь между моделями на уровне Python — позволяет обращаться к связанным объектам как к атрибутам. `cascade="all, delete-orphan"` означает: при удалении родительского объекта автоматически удаляются дочерние (без отдельного DELETE). Например, удаление альбома удалит его треки.

---

**В39. Почему `album_name` хранится в треке, если есть `album_id`?**

Денормализация ради производительности поиска. Поиск по `Track.album_name.ilike(...)` работает без JOIN с таблицей albums. Если бы поле не дублировалось, поиск треков по названию альбома требовал бы JOIN. Компромисс: при изменении названия альбома нужно обновлять `album_name` во всех треках (это делается в `PUT /api/albums/{id}`).

---

**В40. Как реализована связь «многие-ко-многим» между плейлистами и треками?**

Через промежуточную таблицу `playlist_tracks` с тремя ключевыми полями:

- `playlist_id` — FK на playlists
- `track_id` — FK на tracks  
- `position` — порядок трека в плейлисте (0, 1, 2...)

Ограничение `UniqueConstraint("playlist_id", "position")` запрещает два трека на одной позиции. `CheckConstraint("position >= 0")` запрещает отрицательные позиции.

---

**В41. Что такое `server_default=func.now()` и `onupdate=func.now()`?**

- `server_default=func.now()` — PostgreSQL автоматически ставит текущее время при INSERT, Python не участвует
- `onupdate=func.now()` — SQLAlchemy автоматически обновляет поле при UPDATE

Это удобнее, чем вручную задавать `created_at=datetime.now()` при каждом создании объекта.

---

**В42. Как хранятся UUID в PostgreSQL?**

Используется нативный тип `postgresql.UUID(as_uuid=True)` — PostgreSQL хранит UUID эффективно (16 байт) в специальном типе `uuid`. Параметр `as_uuid=True` означает, что SQLAlchemy автоматически конвертирует строку/UUID Python туда и обратно. Для генерации используется `uuid.uuid4()` (случайный UUID v4).

---

**В43. Что такое индексы и какие есть в проекте?**

Индекс — структура данных (обычно B-Tree), ускоряющая поиск. Без индекса PostgreSQL делает полный перебор таблицы (seq scan). В проекте:

- `idx_auth_users_email` — unique-индекс, быстрый поиск при логине
- `idx_tracks_artist`, `idx_tracks_album_id` — для поиска и фильтрации
- `idx_user_profiles_username` — unique-индекс
- Индексы на `listening_history.user_id`, `track_id` — для агрегации истории
- `idx_playlists_owner` — для `GET /playlists/me`

---

**В44. Что произойдёт при удалении трека из playlist_tracks?**

Поскольку `PlaylistTrack` связан с треком через FK с `ondelete="CASCADE"`, при удалении трека из таблицы `tracks` — автоматически удалятся все записи `playlist_tracks`, где этот трек упоминается. SQLAlchemy также настроен через `cascade="all, delete-orphan"` на стороне Python.

---

**В45. Как работает `autocommit=False` в SessionLocal?**

Транзакция начинается неявно при первой операции. `autocommit=False` значит, что каждый запрос не коммитируется автоматически — нужно явно вызвать `db.commit()`. Это даёт контроль: несколько операций можно объединить в одну транзакцию. При ошибке — `db.rollback()` отменяет всё сразу.

---

**В46. Что такое `autoflush=False`?**

По умолчанию SQLAlchemy делает flush (отправляет SQL в БД) перед каждым запросом, даже без commit. `autoflush=False` отключает это. Иногда нужно явно контролировать, когда именно данные попадут в БД. В seed.py используется `db.flush()` перед получением `album.id` (чтобы PostgreSQL присвоил ID), затем `db.commit()`.

---

**В47. Что такое `CheckConstraint` в модели Track?**

```python
__table_args__ = (CheckConstraint("duration > 0", name="check_duration_positive"),)
```

Это ограничение на уровне PostgreSQL: нельзя сохранить трек с длительностью <= 0. Даже если Python-код попытается сохранить такой трек — БД отклонит. Это второй уровень защиты после Pydantic-валидации.

---

**В48. Как SQLAlchemy знает, что делать JOIN при обращении к `album.songs`?**

Через `relationship("Track", back_populates="album_ref")` в модели Album. SQLAlchemy автоматически строит JOIN при обращении к атрибуту. По умолчанию используется **lazy loading** — SQL выполняется только при первом обращении к атрибуту. Это может привести к N+1 проблеме, но для данного проекта объёмы небольшие.

---

**В49. Почему `UserProfile.id` — одновременно PK и FK?**

Это паттерн «Shared Primary Key» для отношения 1:1. `user_profiles.id` совпадает с `auth_users.id` — не генерируется отдельно, а наследуется. Это гарантирует, что никогда не будет профиля без пользователя, и исключает дополнительный индекс. При удалении `auth_users` каскадом удаляется `user_profiles`.

---

**В50. Как `listening_history` используется в рекомендациях?**

При каждом нажатии play на фронтенде — `POST /api/player/play` записывает в `listening_history` (`user_id`, `track_id`, `played_at`). Сервис рекомендаций делает запросы: получает историю слушателя, извлекает уникальных артистов, ищет другие треки этих артистов, которые пользователь ещё не слышал. Это основа rule-based рекомендательной системы.

---

## 5. Alembic — миграции

---

**В51. Что такое Alembic и зачем он нужен?**

Alembic — инструмент управления миграциями БД для SQLAlchemy. Миграция — Python-скрипт с функциями `upgrade()` и `downgrade()`, описывающими изменение схемы БД. Без Alembic при изменении модели нужно вручную писать ALTER TABLE. Alembic хранит историю версий в таблице `alembic_version` и применяет только новые миграции.

---

**В52. В каком порядке применяются миграции в проекте?**

```
001_initial_schema → 003_fk_to_user_profiles → 002_add_music_tables → 004_add_initial_username → 6c885fc11b07_merge_heads
```

Порядок определяется цепочкой `down_revision`. Применяются командой `alembic upgrade head` — до последней версии. В Docker это происходит автоматически при старте backend.

---

**В53. Что создаёт `001_initial_schema.py`?**

Базовые таблицы: `auth_users`, `tracks` (первоначальный вид), `user_profiles`, `playlists`, `playlist_tracks`, `listening_history`. Также активирует расширение PostgreSQL `uuid-ossp` для генерации UUID через `uuid_generate_v4()`.

---

**В54. Что такое `6c885fc11b07_merge_heads.py` и почему он нужен?**

Миграция-слияние. Когда в Alembic две миграции ссылаются на одну предыдущую версию (два `down_revision` указывают на одну) — возникает «разветвление» (`multiple heads`). `alembic upgrade head` выдаёт ошибку. Миграция-слияние объединяет обе ветки в одну. Создаётся командой `alembic merge heads`.

---

**В55. Можно ли откатить миграцию? Как?**

Да, через `alembic downgrade -1` (на одну версию назад) или `alembic downgrade <revision_id>`. Каждая миграция имеет функцию `downgrade()`, которая отменяет изменения. Например, `downgrade` для `002_add_music_tables.py` дропает таблицы `albums`, `messages`, `user_statuses` и удаляет добавленные колонки из `tracks`.

---

**В56. Как добавить новое поле в существующую таблицу через Alembic?**

1. Изменить ORM-модель (добавить Column)
2. Выполнить `alembic revision --autogenerate -m "add field"` — Alembic сравнит модели с БД и сгенерирует миграцию
3. Проверить сгенерированный файл
4. Применить: `alembic upgrade head`

Пример из проекта: `004_add_initial_username.py` добавил поле `initial_username` в `auth_users`.

---

**В57. Почему `alembic upgrade head` запускается при старте контейнера, а не один раз?**

Это стандартная практика: при каждом запуске приложение приводит схему БД к актуальному состоянию. Если контейнер перезапускается после выпуска новой версии с миграциями — они применяются автоматически. Alembic идемпотентен: уже применённые миграции пропускаются.

---

## 6. Бизнес-логика сервисов

---

**В58. Как загружаются файлы? Где хранятся?**

`POST /api/upload/image` и `POST /api/upload/audio` принимают `multipart/form-data`. Файл сохраняется на диск в директорию `media/images/` или `media/songs/`. Имя файла генерируется как `uuid4().hex + extension` — случайная строка без конфликтов. URL возвращается клиенту (`/media/images/abc123.jpg`). В Docker используется named volume `media_data` — файлы переживают рестарт контейнера.

---

**В59. Почему загрузка файлов только для администраторов?**

Загрузка защищена `get_admin_user_id`. Это предотвращает: переполнение диска злоумышленниками, загрузку вредоносных файлов. Проверяется `content_type` файла — принимаются только `image/jpeg`, `image/png`, `image/webp` для изображений и `audio/mpeg`, `audio/mp3` для аудио.

---

**В60. Почему файлы хранятся локально, а не в S3?**

Для учебного проекта локальное хранение проще. В production правильным решением было бы S3 (AWS) или аналог: масштабирование, CDN, надёжность. В текущей архитектуре файлы монтируются через Docker volume и отдаются через Nginx proxy.

---

**В61. Как трек наследует обложку альбома?**

При создании трека через `POST /api/songs`, если указан `album_id`, но не указан `image_url`:

```python
if not data.get("image_url"):
    data["image_url"] = album.image_url
```

Это удобно: при добавлении трека в альбом обложка подставляется автоматически. Администратор может переопределить, передав собственный `image_url`.

---

**В62. Как работает рекомендательный алгоритм?**

Rule-based (без ML) алгоритм в `_compute_recommendations`:

1. Получает треки из истории прослушиваний пользователя → заносит в `exclude_ids`
2. Извлекает топ-10 артистов из истории → ищет другие треки этих артистов (которые ещё не слушал)
3. Берёт треки из плейлистов пользователя (которых нет в истории)
4. Если меньше 20 рекомендаций — добирает новые треки из БД
5. Перемешивает и возвращает максимум 20

---

**В63. Что такое кэш рекомендаций? Как он работает?**

In-memory кэш в виде словаря `_cache: dict[str, tuple[float, list]]`. Ключ — UUID профиля, значение — кортеж (timestamp, список треков). При запросе рекомендаций:

1. Проверяется кэш: если TTL не истёк — возвращается закэшированный результат
2. Если истёк или нет — вычисляется заново, сохраняется в кэш

TTL настраивается через `RECOMMENDATION_CACHE_TTL` (по умолчанию 300 секунд). `threading.Lock()` защищает от race condition при параллельных запросах.

---

**В64. Какие недостатки у текущего кэша рекомендаций?**

- **Не персистентный** — при рестарте приложения кэш теряется
- **Не масштабируется** — если запустить несколько инстансов backend, каждый имеет свой кэш
- **In-process** — занимает память приложения

В production правильное решение — Redis как внешний кэш.

---

**В65. Как устроена история прослушиваний?**

При каждом нажатии play фронтенд отправляет `POST /api/player/play` с `{"track_id": "uuid"}`. Backend проверяет существование трека, создаёт запись `ListeningHistory(user_id, track_id, played_at=now())`. Записи накапливаются — каждое прослушивание фиксируется отдельно. `GET /api/player/history` возвращает историю в порядке убывания времени.

---

**В66. Почему `listing_history` ссылается на `user_profiles`, а не на `auth_users`?**

Архитектурное решение: `auth_users` — это «ключ от системы», `user_profiles` — это «пользователь как личность». История прослушиваний относится к личности, а не к учётным данным. Плейлисты тоже принадлежат профилю. Это разделение позволяет в будущем добавить OAuth без изменения логики плейлистов и истории.

---

**В67. Как обеспечивается порядок треков в плейлисте?**

Поле `position` в `playlist_tracks`. При добавлении трека в плейлист позиция = текущее количество треков в плейлисте (добавляется в конец). Ограничение `UniqueConstraint("playlist_id", "position")` предотвращает коллизии. При сериализации в `PlaylistResponse` треки сортируются по `position` через `sorted(data.tracks, key=lambda pt: pt.position)`.

---

**В68. Что такое `featured` и `trending` треки? Как они отбираются?**

В MVP оба работают одинаково: берут 10 треков из БД и возвращают случайную выборку из них (`random.sample`). В реальном Spotify trending определяется количеством прослушиваний, featured — редакционным выбором. Текущая реализация — заглушка для демонстрации функционала.

---

**В69. Может ли один трек быть в нескольких плейлистах?**

Да. `playlist_tracks` — связующая таблица «многие-ко-многим». Один трек может быть в любом количестве плейлистов, каждый раз с разной позицией. При удалении трека из системы — каскадно удаляются все его записи в `playlist_tracks`.

---

**В70. Как реализована публичность плейлистов?**

Поле `is_public: bool` в модели Playlist. `GET /api/playlists/` возвращает:

- Авторизованным: публичные OR свои (`owner_id == user_id`)
- Анонимам: только публичные

`GET /api/playlists/{id}` проверяет: если плейлист приватный и запрашивает не владелец — возвращает 404 (не 403, чтобы не раскрывать факт существования).

---

**В71. Что проверяет `_require_owner`?**

```python
def _require_owner(playlist, user_id):
    if playlist.owner_id != user_id:
        raise HTTPException(403, "Only the owner can edit this playlist")
```

Защита от того, что один пользователь изменит плейлист другого. Проверяется перед любой операцией записи (PUT, DELETE, добавление/удаление трека).

---

**В72. Как работает `TrackResponse.album_name` — оно не в схеме TrackBase?**

`album_name` хранится в БД (в модели Track), но в схеме `TrackBase` и `TrackCreate` его нет — оно вычисляется на бэкенде из `album_id`. В `TrackResponse` через `from_attributes=True` SQLAlchemy-объект сериализуется напрямую, включая все поля модели. Поэтому `album_name` попадает в ответ API, даже не будучи явно объявленным в схеме ответа.

---

## 7. WebSocket и чат

---

**В73. Что такое WebSocket? Чем отличается от HTTP?**

HTTP — однонаправленный: клиент делает запрос, сервер отвечает, соединение закрывается. WebSocket — двунаправленное постоянное соединение: и клиент, и сервер могут отправлять сообщения в любой момент без нового запроса. Идеально для чата, уведомлений в реальном времени, онлайн-статусов.

---

**В74. Как устроен `ConnectionManager`?**

Singleton-объект `manager` хранит:

- `active_connections: Dict[UUID, List[WebSocket]]` — все активные соединения по user_id (пользователь может иметь несколько вкладок)
- `online_users: Set[UUID]` — set онлайн-пользователей
- `user_activities: Dict[UUID, str]` — текущая активность (например, «Слушает X»)

Методы: `connect`, `disconnect`, `broadcast` (всем), `send_personal_message` (одному).

---

**В75. Как происходит аутентификация WebSocket-соединения?**

WebSocket не поддерживает заголовок `Authorization`. Клиент передаёт токен в параметрах соединения (`auth.token`). При подключении сервер извлекает токен из `websocket.scope["auth"]`, декодирует через `decode_token`. Если токен невалиден — закрывает соединение с кодом 4001.

---

**В76. Что происходит при подключении пользователя к WebSocket?**

1. Токен проверяется
2. WebSocket принимается (`websocket.accept()`)
3. Пользователь добавляется в `active_connections` и `online_users`
4. Статус `is_online=True` записывается в БД (`user_statuses`)
5. Пользователю отправляется `{"type": "connected"}`
6. Всем онлайн-пользователям рассылается обновлённый список онлайн

---

**В77. Как отправить личное сообщение через WebSocket?**

Клиент отправляет JSON `{"type": "send_message", "receiverId": "uuid", "content": "текст"}`. Сервер:

1. Сохраняет сообщение в таблицу `messages`
2. Отправляет получателю `{"type": "receive_message", ...}`
3. Отправляет отправителю подтверждение `{"type": "message_sent", ...}`

Если получатель офлайн — сообщение сохранено в БД и будет получено при следующем `GET /api/users/messages/{user_id}`.

---

**В78. Что происходит при отключении пользователя?**

Перехватывается `WebSocketDisconnect`. Сервер:

1. Удаляет WebSocket из `active_connections`
2. Если соединений больше нет — убирает из `online_users`
3. Обновляет `is_online=False` в БД
4. Рассылает всем `{"type": "user_disconnected", "userId": "..."}`

---

**В79. Что такое `update_activity` в WebSocket?**

Клиент может отправить `{"type": "update_activity", "activity": "Слушает: Song Name"}`. Сервер обновляет `user_activities[user_id]` и рассылает всем `{"type": "activity_updated"}`. Это позволяет показывать в интерфейсе, что слушает пользователь — как в Discord или реальном Spotify.

---

## 8. Docker и деплой

---

**В80. Из каких контейнеров состоит приложение?**

Три контейнера в `docker-compose.yml`:

1. **postgres** — PostgreSQL 16 (alpine)
2. **backend** — FastAPI приложение (Uvicorn)
3. **frontend** — React-приложение за Nginx

---

**В81. Как обеспечивается порядок запуска контейнеров?**

`depends_on` с `condition: service_healthy`. Backend стартует только когда Postgres ответит на `healthcheck` (`pg_isready -U postgres`). Frontend стартует только после backend (`depends_on: backend`). Без этого backend пытался бы подключиться к БД до её готовности.

---

**В82. Что делает `command` в сервисе backend?**

```yaml
command: >
  sh -c "python -m alembic upgrade head &&
         uvicorn app.main:app --host 0.0.0.0 --port 8000"
```

Сначала применяет все непримененные миграции, потом запускает ASGI-сервер. `&&` гарантирует: если миграции упали — Uvicorn не стартует.

---

**В83. Что такое named volumes в Docker? Зачем они нужны?**

```yaml
volumes:
  postgres_data:
  media_data:
```

Named volume — постоянное хранилище Docker, которое не удаляется при рестарте/пересоздании контейнеров. `postgres_data` хранит данные PostgreSQL, `media_data` — загруженные медиафайлы. Без них данные терялись бы при каждом `docker-compose down`.

---

**В84. Как передаются переменные окружения в контейнер?**

Через секцию `environment` в docker-compose. Секретные значения (SECRET_KEY, ADMIN_EMAILS) берутся из `.env` файла проекта через синтаксис `${VARIABLE:-default}`. `.env` файл не коммитится в git (в `.gitignore`).

---

**В85. Почему backend слушает `0.0.0.0`, а не `127.0.0.1`?**

`127.0.0.1` — только loopback (сам контейнер). `0.0.0.0` — все сетевые интерфейсы, включая Docker-сеть. Иначе nginx из контейнера `frontend` не мог бы достучаться до backend по имени `backend:8000`.

---

**В86. Как Nginx проксирует запросы к backend?**

В `nginx.conf` три блока:

- `location /api` → `proxy_pass http://backend:8000` — все API-запросы
- `location /media` → `proxy_pass http://backend:8000` — медиафайлы
- `location /ws` → `proxy_pass http://backend:8000` + `Upgrade: websocket` — WebSocket

`backend` разрешается через Docker DNS по имени сервиса.

---

**В87. Зачем `client_max_body_size 100M` в Nginx?**

По умолчанию Nginx ограничивает тело запроса 1MB. Загрузка MP3-файлов превышает этот лимит — сервер возвращал 413 Payload Too Large. Увеличение до 100MB исправляет эту ошибку. Значение задано как в глобальном `server {}`, так и в `location /api {}` для надёжности.

---

**В88. Почему MP3 исключён из правил кэширования статики?**

```nginx
location ~* \.(js|css|png|jpg|jpeg|gif|ico|svg|woff|woff2|ttf|eot)$ {
    expires 1y;
}
```

MP3 здесь нет намеренно. Если бы mp3 кешировался Nginx как статика — он искал бы файл в директории фронтенда (`/usr/share/nginx/html`), а не проксировал на backend. Аудиофайлы хранятся на backend, поэтому должны проксироваться через `location /media`, а не отдаваться как статика.

## 9. Исправленные баги

---

**В89. Что такое 307-редирект и почему он ломал поиск?**

307 Temporary Redirect — FastAPI по умолчанию редиректит с `/api/search` на `/api/search/` (добавляет слэш). При этом Nginx в Docker менял URL с `http://backend:8000` на внутренний IP контейнера, который фронтенд не видел. Клиент получал редирект на недоступный адрес. Решение: зарегистрировать оба пути (`""` и `"/"`).

---

**В90. Как был исправлен баг с 307 в `songs.py`?**

```python
@router.post("", response_model=TrackResponse)  # без слэша
@router.post("/", response_model=TrackResponse)  # со слэшем
def create_song(...):
```

Оба декоратора вешаются на одну функцию. FastAPI обрабатывает оба URL без редиректа. Аналогичное исправление в `albums.py` и `search.py`.

---

**В91. Почему трек создавался без обложки? Как исправили?**

При создании трека через API клиент передавал только `album_id`, но не `image_url`. Backend создавал трек с пустой обложкой, хотя альбом её имел. Исправление: перед созданием трека проверяется `album.image_url` и копируется в трек, если трек не имеет своей обложки.

---

**В92. Как был исправлен баг с MP3 (аудио не воспроизводилось)?**

Nginx кэшировал запросы к `*.mp3` в правиле статики — пытался найти файл локально, а не проксировать на backend. Решение: убрать `mp3` из regex-паттерна кэширования. Теперь запросы к `/media/songs/...mp3` идут через `location /media` → `proxy_pass backend`.

---

**В93. Почему поиск не находил треки по названию альбома до исправления?**

OR-фильтр в search охватывал только `Track.title` и `Track.artist`. Если пользователь вводил название альбома — треки из него не находились. Исправление: добавлен третий условие `Track.album_name.ilike(f"%{query}%")`. Это возможно благодаря денормализации: `album_name` хранится прямо в треке.

---

**В94. Что такое `pool_pre_ping` и какой баг он предотвращает?**

Без `pool_pre_ping` при долгом простое (например, ночью) PostgreSQL закрывает «спящие» соединения. SQLAlchemy берёт такое соединение из пула и получает ошибку `connection closed`. Запрос падает с 500. `pool_pre_ping=True` делает тихую проверку перед выдачей соединения и при ошибке переподключается прозрачно.

## 10. Безопасность

---

**В95. Как защищены эндпоинты от неавторизованного доступа?**

Через dependency injection: все мутирующие эндпоинты принимают `user_id: UUID = Depends(get_current_user_id)`. Этот dependency не вернёт управление обработчику если токен отсутствует или невалиден — автоматически вернёт 401. Публичные GET-эндпоинты (список треков, альбомов) открыты намеренно.

---

**В96. Как защищены операции записи/удаления контента (треки, альбомы)?**

Дополнительная проверка `get_admin_user_id` — кроме валидного JWT, email пользователя должен быть в списке `ADMIN_EMAILS`. Список задаётся через переменную окружения, не хранится в коде. Обычный пользователь получит 403 при попытке создать или удалить трек.

---

**В97. Защищены ли операции с плейлистами от чужих пользователей?**

Да, через `_require_owner`. После получения объекта плейлиста из БД проверяется `playlist.owner_id == user_id`. При несоответствии — 403. Это реализовано для всех мутирующих операций: PUT, DELETE плейлиста, добавление/удаление трека.

---

**В98. Могут ли пользователи загрузить вредоносный файл?**

Две линии защиты:

1. Проверка `content_type` — принимаются только разрешённые MIME-типы
2. Доступ только для администраторов

Однако нет проверки содержимого файла (magic bytes), что в production является слабым местом. Полная защита требовала бы virus-scan или проверки сигнатуры файла.

---

**В99. Как защищены пароли в БД?**

Пароли никогда не хранятся в открытом виде. Только bcrypt-хеш с индивидуальной солью. Даже если база утечёт — восстановить пароли методом перебора крайне сложно из-за высокой вычислительной стоимости bcrypt. Поле называется `password_hash` — это нигде не возвращается через API (нет в схеме ответа).
